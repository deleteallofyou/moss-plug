# moss-desktop / 小龙虾桌宠开发路线图

> 当前阶段：v0.3 聊天 MVP 已启动并完成第一轮落地
> 
> 目标：先把“状态联动 + 实时交流 + 桌宠体验”做成完整的软件产品，再为未来硬件接入预留协议和架构。

---

## 1. 产品目标

这个项目未来不只是一个“状态看板”，而是一个可以与 OpenClaw 联动的桌宠系统。

长期目标分成三层：

### A. 状态展示层
桌宠可以实时展示小龙虾当前状态：
- 空闲中
- 思考中
- 回复中
- 睡眠 / 离线
- 异常
- 心情 / 场景 / 活跃度

### B. 交互输入层
用户可以通过桌宠和小龙虾互动：
- 点击桌宠
- 文本输入
- 快捷按钮
- 语音输入
- 提醒 / 唤起

### C. 会话通信层
桌宠可以真正作为交流入口：
- 把用户输入发给 OpenClaw
- 接收并展示回复
- 支持流式返回
- 支持多轮对话
- 支持会话绑定

---

## 2. 当前已有基础（截至 v0.2 / 同步修复后）

当前项目已经具备以下基础能力：

### OpenClaw → 桌宠
- 通过 `lobster-status` 插件输出实时状态
- 提供 `status / health / stream` 接口
- 支持 SSE + 轮询回退
- 能根据 OpenClaw 运行态推导桌宠状态

### 桌宠 → OpenClaw
- 能回传轻量设备事件
- 已支持：
  - `page_load`
  - `heartbeat`
  - `visibility_change`
  - `pet_click`
- 已建立设备记录与最近事件记录

### 本地桥接层
- `server.js` 提供本地桥接页面入口
- 当前页面入口：`http://127.0.0.1:8848`
- 可以作为未来桌宠客户端与 OpenClaw 之间的统一访问层

### 当前结论
现在这套结构已经是一个合格的“状态桥 + 设备事件桥”，但还不是完整的“桌宠聊天中台”。

---

## 3. 推荐架构

建议继续保持三层结构：

### Layer 1：OpenClaw
负责：
- 对话能力
- 会话管理
- 记忆 / 工具调用
- 多渠道消息处理

### Layer 2：lobster bridge / plugin
负责：
- 状态统一输出
- 设备事件输入
- 会话转发
- 实时流协议
- 设备注册 / 鉴权 / 兼容层

### Layer 3：桌宠客户端
负责：
- UI
- 动画
- 表情
- 气泡消息
- 文本 / 语音输入
- 本地交互体验

原则：
- OpenClaw 是大脑
- 插件 / bridge 是通信总线
- 桌宠客户端是表现层

# v0.3：桌宠聊天 MVP

## 目标
让桌宠不止能显示状态，还能通过桌宠直接和小龙虾交流。

## 核心能力
- 保留现有状态系统
- 新增桌宠消息发送接口
- 新增会话创建 / 绑定 / 恢复
- 新增流式回复能力
- 新增桌宠聊天 UI
- 状态与聊天联动

## 当前已完成（第一轮 MVP）
- [x] 本地 bridge 增加聊天接口：`/api/chat/history`、`/api/chat/send`、`/api/chat/reset`
- [x] 使用 OpenClaw Gateway helper 在本地 Node 侧转发桌宠消息
- [x] 建立桌宠专用稳定会话 key，并支持“新会话”切换
- [x] 前端加入基础聊天 UI：消息列表、输入框、发送按钮、新会话按钮
- [x] 基础状态联动已打通：发消息后可观察 `thinking / replying / idle` 变化
- [x] README 与项目文档已补充本轮实现说明

## 本轮仍未完成
- [ ] 真正的服务端流式回复（当前 MVP 以历史轮询驱动回复刷新）
- [ ] 更完整的 run 状态追踪
- [ ] 工具调用 / thinking 内容的折叠展示
- [ ] 多会话切换与最近会话列表
- [ ] 更像桌宠产品的交互 / 动画体验

## 必做任务

### 4.1 会话接口
新增接口建议：
- `POST /lobster/session/start`
- `POST /lobster/session/resume`
- `GET /lobster/session/:id`
- `POST /lobster/message`
- `GET /lobster/conversation/stream`

### 4.2 消息协议
统一消息事件类型：
- `user_message`
- `assistant_delta`
- `assistant_final`
- `tool_event`
- `status_update`
- `error`

### 4.3 桌宠聊天 UI
桌面页至少要有：
- 输入框
- 发送按钮
- 消息气泡
- 流式输出区
- 当前状态展示区
- 最近会话区（可选）

### 4.4 状态联动
聊天时桌宠表现应联动：
- 用户发消息 → `thinking`
- 正在输出 → `replying`
- 回复完成 → 回到 `idle`

## v0.3 验收标准
- 用户可以在桌宠 UI 里输入一句话
- 小龙虾可以返回回复
- 支持流式显示或准实时显示
- 状态不会乱卡
- 会话能连续对话

# v0.4：桌宠交互体验完善

## 目标
把“能聊”升级成“好聊”。

## 核心能力
- 动作 / 表情系统
- 多种交互入口
- 更强的状态表现
- 更像宠物而不是普通聊天框

## 必做任务

### 4.5 状态映射规范
固定每个状态的表现：
- 表情
- 动作
- 背景色
- 文案
- 动画速度
- 音效（如有）

### 4.6 输入交互
增加：
- 单击 / 双击 / 长按
- 快捷指令按钮
- 展开 / 收起聊天窗
- 静音 / 休眠按钮

### 4.7 消息体验
- 打字中动画
- 回复中的过渡状态
- 错误提示
- 历史消息折叠 / 清理

## v0.4 验收标准
- 桌宠的状态变化一眼可辨认
- 交互比普通网页更“宠物化”
- 有最基础的沉浸感和陪伴感

让桌宠具备说和听的能力。

## 核心能力
- 语音输入
- 语音输出
- 播放中断

# v0.6：场景系统

## 目标
让桌宠进入“陪伴模式”，不是只有聊天时才存在。

## 建议场景
- 工作模式
- 摸鱼模式
- 夜间安静模式
- 提醒模式
- 陪伴模式
- 忙碌模式

## 必做任务

### 4.11 场景来源
场景可由以下条件触发：
- 时间段
- OpenClaw 当前状态
- 日历 / 提醒
- 用户手动切换
- 设备在线情况

### 4.12 场景输出
场景应影响：
- 表情
- 文案
- 动画节奏
- 提醒行为
- 语音风格（未来）

## v0.6 验收标准
- 桌宠在不聊天时也有“存在感”
- 状态与场景不会互相冲突

# v0.7：多设备 / 多终端支持

## 目标
为未来小屏、悬浮窗、控制页、硬件桌宠做好统一设备管理。

## 必做任务

### 4.13 设备注册
每个设备需要明确：
- `deviceId`
- `deviceName`
- `platform`
- `clientVersion`
- `capabilities`

### 4.14 设备能力抽象
建议定义能力：
- `display`
- `audio_out`
- `mic_in`
- `touch`
- `led`
- `motor`

### 4.15 多设备状态策略
需要设计：
- 主设备 / 从设备
- 广播哪些状态
- 哪些设备可发消息
- 哪些设备只看状态

## v0.7 验收标准
- 同一个小龙虾可以被多个设备消费
- 不同设备有不同能力但共用协议

## 7. 风险与注意事项

### 技术风险
- 状态源混乱导致表现失真
- 不同客户端事件协议不一致
- SSE 断线或轮询兜底不完整
- 会话上下文丢失
- 语音输入输出时序打架

### 产品风险
- 只会显示状态但不会聊天，用户会觉得弱
- 聊天能用但体验像网页，不像桌宠
- 功能多但没有统一协议，后期硬件接入会非常痛苦

### 原则
- 先统一协议，再堆功能
- 先完成 MVP 闭环，再做高级体验
- 先把软件版做完整，再接硬件
